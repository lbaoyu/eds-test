AEM 6.5 LTS Upgrade – Technical Component Change Catalog

This document summarizes the main technical changes when upgrading from an existing Adobe Experience Manager (AEM) 6.5 deployment (using standard service packs) to the current AEM 6.5 Long-Term Support (LTS) release.

Scope. This overview focuses on platform and runtime changes – Java, servlet engine, repository, dispatcher, and key AEM feature deprecations. It is intended as customer‑facing upgrade background, not as a step‑by‑step runbook.

1. High‑Level Overview

What this LTS upgrade is:

  • A modernization of the AEM 6.5 runtime, not a new functional major version.
  • Moves AEM to a newer long‑term‑support Java (Java 21) and servlet engine.
  • Upgrades core repository components (Oak, Sling, OSGi) and dispatcher.
  • Removes deprecated features and vulnerable 3rd‑party libraries.

Typical goals:

  • Extend the supported lifetime of the existing 6.5 codebase.
  • Improve security posture and performance (JDK + dependency updates).
  • Align with Adobe’s forward‑looking authoring and headless capabilities.

2. Runtime & Operating System Layer

Operating systems themselves do not fundamentally change with the LTS upgrade, but the Java version and some low‑level expectations do.

Area Before (typical AEM 6.5) After (AEM 6.5 LTS) Notes
Operating System Enterprise Linux / Windows versions as per original 6.5 system requirements. Same OS families, provided they support Java 21. Most changes are at the JVM and application layer. OS must meet current AEM 6.5 LTS system requirements.
Java (JDK / JVM) Java 8 or Java 11 (depending on initial deployment). Java 21 (LTS). AEM LTS is designed to run on Java 21. Old flags such as -XX:PermSize / -XX:MaxPermSize must be removed. Runtime and GC behavior are those of Java 21.
JVM GC & Tuning Older GC defaults (e.g., Parallel GC / CMS) and tuning practices. Modern GC options such as G1 or ZGC are recommended. Heap usage profiles often improve; GC logs and tuning should be revisited post‑upgrade.

3. AEM Core & Repository Stack

Component Before (standard AEM 6.5) After (AEM 6.5 LTS) Impact / Considerations
AEM Distribution 6.5 GA + one or more service packs (e.g., 6.5.10, 6.5.16, etc.). 6.5 LTS (also delivered as an updated 6.5 quickstart). Consolidates multiple years of service pack fixes plus compatibility and security updates into a single long‑term‑support baseline.
Servlet Container (Quickstart) Earlier Jetty‑based servlet engine (e.g., Jetty 9.x). Updated Jetty‑based servlet engine (Jetty 11.x line). Improved TLS/security and HTTP behavior; low‑level servlet and filter behavior should be regression tested.
Apache Sling & OSGi (Felix) Versions aligned with the existing 6.5 service pack. Newer Sling and Felix versions. Generally transparent; uncommon OSGi wiring patterns may need attention.
Oak (JCR Repository) Oak 1.4x line (exact version depends on SP). Higher Oak 1.6x line. Brings fixes and performance improvements. Custom indexes should be re‑validated; some definitions may evolve.
Node Store TarMK (segment store) for most 6.5 deployments. TarMK remains the standard. No structural change; the upgrade runs repository migration tasks in place.
Data Store FileDataStore, Amazon S3, or Azure Blob Store using older connector bundles. Same storage options, using updated connector bundles. Connector versions for S3 / Blob are updated; configuration should be checked, but data layout remains compatible.
Quickstart Size & Layout Larger quickstart including several legacy libraries. Smaller quickstart; unused and deprecated libraries removed. Reflects removal of old dependencies such as Guava and legacy collections libraries from the platform image.

4. Web Tier, Dispatcher, and External Integrations

Component Before After Impact / Considerations
Dispatcher Module Previous 4.3.x dispatcher version. Dispatcher 4.3.x updated to the latest available line. Includes security and compatibility improvements and an updated SSL module. Dispatcher configuration should be re‑validated.
Web Server Apache HTTP Server 2.4.x (or equivalent supported servers). Same generation, with updated configuration recommendations. Existing virtual host and module configurations should be reviewed and regression‑tested with the updated dispatcher.
Cloud Storage Connectors Older S3 / Azure Blob connector versions. Newer S3 / Azure Blob connector versions. Configuration model is similar, but connector bundle versions change; this should be part of upgrade testing.

5. AEM Product & Feature Layer

5.1 Enhancements and Modern Capabilities

The LTS line consolidates existing 6.5 features and prepares for newer authoring and headless scenarios. Typical benefits include:

  • Improved performance and stability from newer Java, Oak, and servlet engine.
  • Better security posture due to updated platform libraries and removal of vulnerable ones.
  • Alignment with modern AEM authoring experiences (such as Universal Editor and headless content delivery) where licensed and applicable.

5.2 Deprecated and Removed Features

The LTS upgrade explicitly discontinues or removes several older feature sets and modules:

Area Status in AEM 6.5 LTS Impact on Implementation
Communities / Social No longer supported; related bundles and content are removed or deprecated. Any custom code, components, or integrations that relied on legacy Communities APIs or JCR structures must be refactored or retired.
Screens No longer supported. Screen‑based experiences built on the legacy Screens framework must be replaced with other approaches.
Classic Commerce / Older CIF Integrations Deprecated or removed in favor of newer commerce approaches. Classic AEM commerce nodes, overlays, and resource types should be migrated, replaced, or removed before or as part of the upgrade.
Older SPA Editor Flows Superseded by the Universal Editor direction. Existing SPA Editor integrations may need to be validated and adjusted to ensure they continue working or to migrate to newer patterns.
Legacy Portal / Portal Director No longer supported. Custom portal‑style integrations built on these features should be considered for redesign.
Various Legacy DAM/Utility Bundles Removed where superseded or unused. If custom code directly referenced these bundles, alternate approaches or APIs will be required.

6. Third‑Party Libraries & API Changes

One of the important aspects of the LTS upgrade is cleaning up third‑party library dependencies that are outdated or have known vulnerabilities.

Library / API Change in LTS Impact on Custom Code
Google Guava The Guava library is no longer shipped as a platform‑wide dependency within AEM. Any custom bundles that relied on the Guava version previously embedded in AEM should declare and include their own Guava dependency at a supported version. This should be updated and managed as part of the customer’s build.
Apache Commons‑Collections 3.x Legacy commons‑collections 3.x APIs are removed due to vulnerability and obsolescence. Custom code using these APIs should migrate to newer collections APIs or alternate libraries. Compilation and runtime testing should be used to ensure no remaining references.
Servlet API Servlet API alignment with the newer Jetty version (Jakarta‑aligned APIs). Custom servlets and filters that assume older or tightly version‑pinned servlet APIs should be checked and, if necessary, adjusted to match the newer servlet container behavior.
Java SE APIs Running on Java 21 means some older Java APIs and practices are discouraged or no longer available. Certain internal or removed APIs may no longer be usable. Custom code should be compiled and tested on Java 21. Static analysis and migration guides should be used to identify any use of removed or strongly discouraged Java APIs.
Legacy Scene7 APIs Some older Scene7‑specific Java APIs are treated as deprecated or unsupported. Custom Scene7 integrations may need to align to newer integration points or APIs.

7. Indexing & Repository Behavior

The underlying indexing model (Oak indexes in the repository) remains the same, but versions and behavior are updated.

Aspect Before After Impact / Considerations
Oak Version & Index Implementation Oak 1.4x line with corresponding index behavior. Oak 1.6x line with updated index behavior. Newer Oak versions can introduce optimizations and subtle behavior changes. Custom indexes under /oak:index should be validated and workloads that rely heavily on queries should be performance‑tested after upgrade.
Index Definitions Customer‑managed definitions under /oak:index. Same model, with updated core index definitions from AEM. Some indexes may be added or adjusted by the upgrade. It is recommended to check for any reindex flags and resolve them.
Text Extraction & DAM Indexing Indexing of assets via the existing Tika/Oak pipeline. Same approach, with updated libraries. For large DAM implementations, indexing performance and timings should be monitored closely after upgrade.

8. Operational Behavior & Tooling

The LTS release also formalizes and emphasizes certain operational practices around AEM upgrades.

Area Before After Impact / Considerations
Upgrade Tooling In‑place upgrade procedure, pre‑upgrade checks, and basic pattern detection. Same tools, plus more formalized compatibility checks and analyzer reports. It is recommended to run AEM’s pre‑upgrade health checks, pattern detectors, and any code analyzers against the customer codebase to identify deprecated feature usage and incompatible APIs before upgrading.
Monitoring & Logging Logback‑based logging and GC logs for Java 8/11. Same logging model, but with Java 21 runtime characteristics. GC behavior and heap profiles may differ; operations teams should review and tune JVM options post‑upgrade.
Maintenance Tasks Offline revision cleanup, datastore garbage collection, workflow/audit/version purges. These remain required and are explicitly called out as pre‑upgrade and ongoing best practices. Running these tasks prior to upgrade typically reduces upgrade time and risk, especially on large repositories.

9. How to Use This Catalog

This document is intended as a structured reference for planning and risk assessment. In practice, we recommend:

  • Using these tables as a checklist of areas to validate (Java 21 readiness, use of removed features, dependency cleanup).
  • Capturing which items apply to your implementation (for example, whether social, Screens, or legacy commerce were ever used).
  • Feeding this into an upgrade test plan and communication plan for development, operations, and business stakeholders.

For a detailed, environment‑specific runbook (commands, exact versions, and timelines), this catalog should be combined with an assessment of your current AEM version, custom codebase, and integrations.
