Architecture Positioning

One storefront, multiple layers, clearer responsibilities

The strongest architecture for KrisShop is not an "EDS or React" decision. It is a layered model: EDS for experience-led pages, Commerce Storefront for commerce-led journeys, React for app-like modules, SSR for discoverability, and LLMO for AI-era product visibility.

View the ownership model See what Commerce Storefront adds

What this architecture solves

  • Separates content-led, commerce-led, and app-like experiences cleanly
  • Avoids forcing one frontend pattern across every page type
  • Supports phased modernization instead of a risky big-bang rewrite
  • Improves readiness for both classic search and AI-driven product discovery

Why this decision matters

KrisShop is not only selecting a frontend technology. It is deciding how one storefront should support multiple experience modes without becoming slow, hard to author, difficult to modernize, or opaque to search and AI systems.

Experience-led

Content agility

Campaigns, storytelling pages, seasonal collections, and homepage modules need speed and fast publishing workflows.

Commerce-led

Discovery and conversion

Search, PLP, PDP, cart, and checkout need product-aware behavior, storefront consistency, and merchandising alignment.

App-like

Rich interaction

Account tools, loyalty modules, dashboards, and utility flows need stronger client-side state and workflow logic.

AI-era

Machine-readable meaning

High-value pages need to be understandable not only by people, but also by search crawlers and AI agents.

One clear ownership model

The most important architectural decision is to map page intent to the right layer once, clearly, and consistently.

Experience area Typical examples Best-fit layer Why it fits
Experience-led Homepage, seasonal campaigns, branded landing pages, editorial and storytelling pages EDS Best for speed, content velocity, experimentation, and experience-rich authoring
Commerce-led Search, PLP, PDP, minicart, cart, checkout Commerce Storefront Best for product discovery, storefront behavior, and conversion-oriented flows
App-like Loyalty tools, account utilities, custom dashboards, workflow-heavy modules React Best for rich interaction, local state, and task-oriented authenticated UX
Discoverability-critical High-value PDP and PLP rendering for search and AI interpretation SSR / pre-render + LLMO Best for semantic visibility, rendered product meaning, and machine readability

What each major layer contributes

The hybrid model works because each layer solves a different problem well. The goal is not one universal framework. The goal is to avoid asking the wrong technology to solve the wrong problem.

EDS

Experience delivery layer

EDS is the right answer when the main challenge is to create and publish high-impact content quickly.

  • Strong for campaigns, landing pages, storytelling, and fast-changing homepage content
  • Supports a lightweight, performance-first page model
  • Improves authoring velocity for content-led surfaces
  • Acts as the experience shell, not the full commerce layer
Commerce Storefront

Commerce execution layer

Commerce Storefront turns a fast frontend into a real storefront that can support discovery, evaluation, and conversion.

  • Owns search, listing, product detail, cart, and checkout behavior
  • Aligns rendering to products, categories, pricing, and merchandising
  • Provides a stronger path for commerce modernization than custom-building everything
  • Sits between the experience layer and commerce services
React

Application behavior layer

React should be used where the interface behaves more like an application than a page.

  • Best for dashboards, loyalty flows, profile centers, and utilities
  • Well suited to local state, task workflows, and rich interaction
  • Powerful for specialized tools
  • Too heavy to be the default answer for every storefront page

What Commerce Storefront has to offer

Commerce Storefront is more than a set of page templates. It is the layer that brings commerce-native behavior, product-aware rendering, and structured extensibility into the storefront.

Commerce-native building blocks

Commerce Storefront gives KrisShop a structured starting point for core shopping journeys.

  • Product listing experiences
  • Product detail experiences
  • Minicart and cart behavior
  • Checkout flows
  • Authentication and account-adjacent journeys

Product-aware rendering

It is designed around the mechanics of shopping rather than generic page rendering.

  • Category and product-driven layouts
  • Pricing-aware rendering
  • Search and discovery behavior
  • Merchandising-aligned experience patterns

Alignment to commerce services

It does not only render pages. It participates in the commerce operating model.

  • Catalog service alignment
  • Product discovery and search alignment
  • Recommendation support
  • Merchandising service integration

Extensibility without full reinvention

KrisShop can adapt storefront behavior without rebuilding the entire commerce surface in custom UI code.

  • Slots and extension points
  • Styling tokens and CSS customization
  • Content enrichment patterns
  • App Builder-based orchestration where needed

Better delivery economics

Using storefront primitives changes where engineering effort goes.

  • Less time spent recreating baseline shopping behavior
  • More time spent on differentiated experiences
  • Cleaner path for premium merchandising and curated collections
  • Lower tendency to solve everything as a custom React project

Stronger fit for modernization

It lets KrisShop modernize commerce-facing surfaces without redesigning the whole digital estate at once.

  • Supports incremental rollout
  • Improves discovery surfaces first
  • Preserves flexibility for backend evolution
  • Creates a more structured target architecture

Bottom line

Commerce Storefront should be understood as the commerce execution layer: it gives KrisShop a structured way to modernize discovery and conversion journeys without reducing every commerce page to a custom frontend build.

Why hybrid beats a single-stack answer

The question should not be "Which one frontend wins?" The better question is "Which layer should own which part of the storefront?"

Approach What it does well Where it becomes limiting
EDS-only Content velocity, performance-first experience pages, lightweight publishing Does not by itself solve product-aware storefront behavior, search/discovery depth, or full commerce execution
React-only Rich interaction, stateful tools, custom UI composition Heavier for performance, harder to govern for discoverability, and more developer-dependent for content-led surfaces
Commerce-only Transactional and shopping flows Too narrow for experience-led storytelling, fast content iteration, and broader marketing ownership
Hybrid model Assigns the right layer to the right job Requires clearer governance, but creates a much stronger long-term storefront architecture

Why phased modernization works best

Frontend modernization should be treated as a phased program by surface area, not a big-bang rewrite. That creates faster value, lower risk, and better organizational alignment.

Phase Focus Best-fit layer Why this phase creates value
Phase 1 Homepage, campaigns, landing pages EDS Fastest way to improve speed, publishing agility, and visible brand experience without touching deeper commerce logic first
Phase 2 Search, PLP, PDP Commerce Storefront Modernizes the highest-value product discovery surfaces where shoppers evaluate and compare products
Phase 3 Cart, checkout, shopper journeys Commerce Storefront Moves deeper into conversion-critical paths after the storefront foundation and key integrations are proven
Phase 4 Specialized tools, utilities, workflow-rich modules React where needed Adds advanced interaction without making the whole site heavy or app-like by default

Why phased is stronger

  • Lowers delivery risk
  • Accelerates visible value
  • Allows side-by-side operation
  • Avoids one migration timeline for every team

What it avoids

  • A full rewrite mentality
  • Unnecessary rebuilding of stable areas
  • Over-using React where simpler layers already fit
  • Delaying value until the whole estate is replaced

What it enables

  • Incremental learning
  • Cleaner governance by page type
  • Easier performance tuning
  • Stronger alignment between content and commerce teams

How to position the commerce layer commercially

The discussion should not be framed as "EDS versus React." It should be framed as a layered model where the commerce-led surface gets a proper commerce operating layer.

Correct framing

  • What owns the experience layer? EDS
  • What owns the commerce layer? Commerce Storefront
  • What owns app-like workflows? React where needed

Best-fit packaging

The cleanest commercial position for the commerce layer is Adobe Commerce Optimizer, because it brings together Commerce Storefront on Edge Delivery, catalog and merchandising services, product discovery direction, recommendation support, and App Builder extensibility.

Why discoverability must be designed in

For high-value commerce pages, discoverability should be treated as an architecture rule, not an optional enhancement. Important meaning needs to be present in the first response, not only after browser execution.

Why SSR matters

  • Improves search engine understanding
  • Supports crawlability and indexing reliability
  • Exposes product and category meaning earlier
  • Creates stronger semantic output for AI agents
  • Reduces reliance on hydration for basic discoverability

Design principle

Render the meaning first. Hydrate the behavior second.

EDS already supports strong server-rendered delivery for experience-led pages. Commerce-led pages should render their important product meaning early, while hydration is reserved for dynamic behavior such as personalization, cart interaction, and workflow logic.

Must exist in rendered HTML Why it matters
Title, meta description, canonical, hreflang Supports page interpretation, deduplication, internationalization, and indexing
H1 and semantic section structure Improves machine-readable clarity and content hierarchy
Product or category descriptions Helps search systems and AI agents understand what the page is about
Breadcrumbs and internal navigation Strengthens site structure, crawlability, and relationship between pages
Product names and image URLs Improves relevance and discoverability in product-oriented experiences
Structured data for product, list, and breadcrumb entities Supports search interpretation and machine-readable understanding of product context

How LLMO changes the target

LLMO changes the architectural target from "Can the page load fast?" to "Can the page be understood correctly by AI systems?"

Agent-ready PDP

Rendered product meaning

The product page should expose enough structured, rendered information for AI agents to understand, compare, and accurately cite the product.

Agent-ready catalog

Catalog enrichment

The catalog should contain stronger names, descriptions, and metadata so products can be found through conversational and attribute-led discovery.

Optimize at Edge

Agent-facing output

Optimize at Edge complements the model by improving AI-facing output without forcing changes to the human-facing storefront experience.

What LLMO requires from the storefront

Stable semantic HTML Complete product meaning Useful rendered metadata Recoverable product details Catalog enrichment Machine-readable clarity

Final position

  • EDS for experience-led pages
  • Commerce Storefront, best packaged through Commerce Optimizer, for commerce-led journeys
  • React only where the experience is genuinely app-like
  • SSR as a discoverability rule for high-value commerce pages
  • LLMO for AI-era product visibility

  • Faster experience-led surfaces
  • Stronger product discovery and shopping flows
  • Clearer division of frontend responsibilities

  • Better readiness for both classic search and AI-led discovery
