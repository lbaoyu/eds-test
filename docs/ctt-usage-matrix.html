Content Transfer Tool (CTT): When You Should and Should Not Use It

This guide summarizes common CTT (Content Transfer Tool) scenarios for migrations to AEM as a Cloud Service (AEM CS), including where CTT is recommended, constrained, or explicitly out of scope.

1.Scenarios where you should use CTT

# Scenario Use CTT? Why / preferred approach
1 On-prem → AEM CS (initial migration) Yes Primary CTT use case: migrate from AEM 6.3+ on-prem to AEM CS, including pages, assets, users/groups, and (within limits) versions. Plan a full migration plus top-ups using the Content Transfer Tool.
2 On-prem → AEM CS (incremental / top-up) Yes Fully supported via CTT “top-up extraction” and “top-up ingestion” (wipe=false) on the same source instance and migration set. Used to shorten the final content-freeze.
Reference: Overview to Content Transfer Tool – Top-Up
3 AMS → AEM CS (initial migration) Yes Officially recommended for AMS → CS: install CTT on AMS Author/Publish and migrate Author→Author and Publish→Publish.
Reference: Migration - AEMaaCS - Content Transfer Tool - AEM on-premise to AEMaaCS (CTT)
4 AMS → AEM CS (incremental / top-up during cut-over) Yes Supported and recommended: frequent CTT top-ups from AMS Author to CS Author (wipe=false) until final cut-over.
Reference: AEM as a Cloud Service content migration FAQ
5 On-prem/AMS → multiple AEM CS envs (Dev, Stage, Prod) Yes One extraction, many ingestions: CTT + CAM migration sets can be ingested into several AEM CS environments from the same source run.
Reference: Implementation Phase in Cloud Acceleration Manager
6 On-prem/AMS → AEM CS Publish only Yes (with care) Supported to install CTT on source Publish and ingest directly to CS Publish, but CTT does not distinguish “published vs unpublished” content. Plan Author and Publish migrations carefully (often Author→Author plus replication is safer).
Reference: Running the Content Transfer Tool on a Publish Instance
7 On-prem/AMS → AEM CS, moving only DAM assets (from AEM) Yes (often best) If assets already live in AEM, CTT is fine and preserves metadata, renditions, and versions where supported. For non‑AEM sources, prefer Assets Bulk Import Service.
Reference: AEM as a Cloud Service content migration FAQ

2.Edge / constrained CTT usage

# Scenario Use CTT? Why / notes
8 On-prem/AMS → AEM CS, >~10 TB total repository “Yes, but only with engineering oversight” CTT can technically work, but documentation flags >~10 TB as a risk zone; you’re expected to engage Adobe before attempting. May require content slicing, special infrastructure, and very careful planning.
Reference: CTT :: Questionnaire to determine usability
9 On-prem/AMS → AEM CS, content under immutable paths (/etc, /apps, /libs) “Partially, but generally No for those paths” CTT doesn’t move immutable areas; those must be restructured (e.g. repository restructuring) and deployed as code. Use CTT for mutable content only; use deployment and refactoring for code and legacy /etc.
Reference: CTT :: Questionnaire to determine usability

3.Scenarios where you should not use CTT

Everything in this section is explicitly out of scope for CTT.

Key reference: Does CTT (Content Transfer Tool) work between cloud instances? “For CTT to migrate existing content to the target AEM Cloud Service instance, there must be a non‑cloud AEM source instance… therefore, no, CTT does not work between cloud instances.”

# Scenario Use CTT? Why / alternative
10 Stage → Prod (AEM CS → AEM CS, same program) No CTT cannot run on AEM as a Cloud Service nor perform CS→CS copies. Stage→Prod promotion is intentionally not a standard pattern. Use AEM packages (scripted/automated) or a one‑off CSME/engineering‑led operation for large moves; target a steady state where authors work directly in Prod.
11 Dev → Stage (AEM CS → AEM CS, same program) No (CTT) CTT doesn’t support CS→CS. A limited Dev→Stage flow is supported via the Content Copy Tool (forward flow) for mutable paths only.
Reference: The Content Copy Tool
12 Prod → Stage / Dev / RDE (AEM CS → AEM CS, same program) No (CTT) Supported route is Content Copy Tool via Cloud Manager (Prod→lower), not CTT. CTT cannot be installed on AEM CS as a source.
Reference: The Content Copy Tool
13 CS → CS across programs (Program A → Program B) No CTT only works from on‑prem/AMS; there is no built‑in cross‑program CTT or Content Copy. Internal guidance is that packages (plus custom tooling or CSME help) are the only general option today.
Reference: Cloud Service to Cloud Service Migrations
14 CS Dev/Stage → CS Prod using CTT No Same limitation: CTT cannot be used CS→CS. For lower→Prod population you’re back to packages or bespoke services. Product guidance remains: “author in Prod, use Content Copy only for Prod→lower.”
15 Non‑AEM source (S3/Azure) → AEM CS No (CTT) CTT expects an AEM repository as source. For S3/Azure buckets, use Assets Bulk Import Service.
Reference: How to ingest and process assets into Experience Manager Assets as a Cloud Service
16 Local filesystem / 3rd‑party CMS → AEM CS directly No (CTT) This is not an AEM→AEM migration. Use Bulk Import Service, the aem-upload client, or custom ingestion via HTTP APIs.
17 Prod → Stage on AMS (not CS) No (CTT) CTT is intended for AMS/on‑prem → AEM CS, not intra‑AMS moves. For AMS Prod→Stage, use Cloud Manager Content Copy for AMS (where available) or AMS‑documented package/VLT patterns.
18 Consolidation / split of CS programs (e.g., “move from normal to Enhanced Security program”) No This is a known tooling gap. Multiple customers have done this only via packages, backflow prototypes, or offsite restore. CTT is explicitly “not supported on Cloud to Cloud”.
Reference: Deloitte – Cloud Service to Cloud Service Migration

4.Summary

  • Use CTT for on‑prem/AMS → AEM CS migrations of mutable content (pages, assets, users, groups).
  • Use with caution for very large repositories (>~10 TB) and remember it doesn’t handle immutable paths.
  • Do not use CTT for any Cloud→Cloud moves or non‑AEM sources; rely on Content Copy Tool, packages, or Bulk Import instead.
