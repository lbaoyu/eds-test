Handling Broken Links & Accessibility on On‑Prem AEM (without Sites Optimizer)

For an on‑prem AEM 6.5 customer, AEM Sites Optimizer is not available. Instead, we can combine AEM’s built‑in capabilities with external tooling to improve both broken‑link handling and accessibility directly in the authoring flow and before the first publish.

1. Broken Links – From Manual to Tool‑Assisted

What AEM Provides

  • AEM’s built‑in Link Checker (on author):
    • Validates internal links as authors add them. If a target page is removed, the link is stripped from publish HTML and shown as broken in author.
    • Tracks and validates external links asynchronously and lists them in the External Link Checker console.

Recommended Pattern for On‑Prem

  • Keep Link Checker enabled on author, tuned for performance
    • Configure via /system/console/configMgr (Day CQ Link Checker Service / Task / Transformer).
    • Use override patterns or x-cq-linkchecker="skip" for domains that cause false positives or slow checks.
  • Disable or minimize Link Checker on publish
    • Avoid performance overhead on the public site; rely on author‑time checks plus external crawlers.
  • Introduce a scheduled external crawler for the live site
    • Use a crawler (e.g. Screaming Frog or an internal tool) to regularly scan the public URLs.
    • Generate reports of 4xx / 5xx broken links and their referring pages for content teams to fix.
Result: Authors catch most broken links during authoring, and SEO/ops teams find any remaining issues via automated live‑site scans instead of manual clicking.

2. Accessibility – Closer to “Compliant on First Publish”

AEM on‑prem does not include a full WCAG validator, but we can push accessibility earlier into the process:

(a) Accessibility‑Aware Components & Templates

  • Prefer and update Core Components to output semantic, accessible HTML.
  • Enforce a proper heading hierarchy and landmarks (single <h1>, clear <main>, <nav>, etc.) at the template level.

(b) Author‑Time Rules Inside AEM

  • Make alt text mandatory on image components, or provide an explicit “decorative image” toggle.
  • Use policies and dialog validation to require labels, descriptions, and other fields that impact accessibility.
  • Restrict risky styling (e.g. free‑form colors that could break contrast) via the Style System/policies.

(c) Self‑Service Accessibility Checks in Authoring

  • Standardize on browser tools (e.g. axe, WAVE, Lighthouse) that authors run on AEM’s Preview/View‑as‑Published URLs.
  • Turn this into a simple pre‑publish checklist for important page types: run scan → fix issues → then request activation.

(d) Optional: Automated Checks in CI/CD

  • If they have Jenkins/GitLab/etc., integrate Pa11y, axe‑core CLI or Lighthouse CI to scan a small set of critical URLs.
  • Fail the build or flag it whenever accessibility violations exceed an agreed threshold.
Result: Most accessibility problems are prevented by design (components/templates), visible to authors during editing, and reinforced by automated checks on key journeys.

3. How to Position This to the Customer

Since they are on‑prem and cannot use AEM Sites Optimizer, position the solution as:

  • Broken links: use AEM Link Checker on author plus a scheduled external crawler on publish to remove manual, ad‑hoc checking.
  • Accessibility: design accessible components/templates, enforce key fields in AEM dialogs, and require quick accessibility scans in preview before first publish, optionally backed by CI‑level checks.

Together, these changes move both broken‑link handling and accessibility into the authoring and deployment flow, giving them much closer to a “correct on first publish” experience on on‑prem AEM.
