Design High‑Value LLMO Prompts for AIS

Using a 4‑step framework to turn real AIS customer questions into a curated prompt catalog and a scalable LLM Optimizer footprint.

1. What This Session Is (and Isn’t)

1.1 This session is about

  • Using a 4‑step framework to design Categories, Topics and Prompts for AIS.
  • Turning real AIS customer questions from search, app and support into a curated prompt catalog.
  • Landing a Phase‑1 prompt footprint (~1,500 prompts) across key AIS journeys (mobile, AIS Fibre, roaming, billing, rewards).

1.2 This session is not about

  • Deep technical details of LLM Optimizer configuration or model tuning.
  • GEO architecture, integration or analytics dashboard training.
  • Choosing or evaluating specific LLMs, search engines or ranking algorithms.

2. Why Prompt Design Matters for AIS

Even with the right tool like LLM Optimizer, GEO outcomes for AIS depend heavily on the design of Topics and Prompts:

  • What we track – the Categories, Topics and Prompts we choose, which define the scope of optimisation.
  • How well prompts reflect real questions AIS customers actually ask across channels (Google, ais.th, myAIS app, contact centre).
  • Whether prompts focus on high‑demand, high‑impact intents (e.g. bills, data usage, roaming, fibre installation) rather than every theoretical question.
  • How clearly we map prompts to owners (Mobile, AIS Fibre, Digital, Care), journeys and KPIs (call deflection, app self‑service, NPS, complaint reduction).
  • How efficiently we use prompts – LLMO charges by prompt volume, so unfocused lists increase cost, noise and maintenance effort without adding insight.
Key idea: Good prompt design makes GEO insights explainable, actionable and tied to AIS’ real customer journeys and KPIs.

4. Foundations: Constraints & Shared Vocabulary

4.1 Constraint: We Don’t See AI Conversations

Tools like ChatGPT, Gemini, Perplexity and Copilot:

  • Do not expose brand‑specific chat histories for AIS.
  • Do not offer a feed like “all prompts about AIS mobile plans this month”.

Implications for AIS

  • We cannot build prompts by exporting AI chat logs.
  • We must base Categories, Topics and Prompts on:
    • Google Search and other search data where AIS appears.
    • On‑site search on ais.th and the myAIS app.
    • Support, chatbot and contact centre logs.
    • Internal expertise from product, network, digital and care teams.
  • LLM Optimizer gives AIS a new lens on AI answer quality, so prompts should reflect:
    • Real customer questions, in context (e.g. prepaid vs postpaid, AIS Fibre vs mobile).
    • End‑to‑end journeys (e.g. sign‑up, usage, billing, cancellation).

4.2 Where We See Real Question Patterns

Google Search Console (GSC) & external search

  • Queries where “AIS” or ais.th appears in search results.
  • Strong signal of what people type into Google/Bing, including:
    • “ais roaming charges japan”, “ais fibre installation time”, “ais prepaid add data”.

On‑site and in‑app search

  • Searches on ais.th and within the myAIS app.
  • Often more product‑ and account‑specific, e.g. package names, add‑ons, data rollover, billing cycles.

Support & chatbot logs, FAQ & Help content

  • Contact centre and chat logs reveal recurring complaints and confusion points.
  • FAQ and Help content reflect curated, official answers and are useful for:
    • Validating prompt wording.
    • Checking coverage across journeys (e.g. SIM activation → usage → bill payment).

4.3 Shared Vocabulary: Category → Topic → Prompt

To stay organised and scalable, we use this hierarchy:

  • Category – high‑level AIS business domain.
    • Examples: Mobile plans & usage, AIS Fibre broadband, Roaming & international, Billing & payments, Rewards & loyalty.
  • Topic – cluster of related customer intents within a Category.
    • Examples: “Prepaid top‑up & data packages”, “Fibre installation & appointment booking”, “Roaming rates & activation”.
  • Prompt – one canonical natural‑language question representing a realistic customer intent.
    • Each Prompt in LLMO stands for a cluster of similar ways customers ask the same thing.

Example

  • Category: Mobile plans & usage
  • Topic: Prepaid top‑up & data packages
  • Prompt: “How can I check my AIS prepaid data balance and buy more data in the app?”

4.4 Intent Themes in Design (Optional)

Intent Themes are recurring question patterns within a Topic:

  • Category: Roaming & international
  • Topic: Roaming rates & activation
  • Intent Theme (examples):
    • Activation & eligibility.
    • Rates by destination.
    • Bill shock & usage control.
  • Prompts: individual questions mapped under each Intent Theme.

Why use Intent Themes in the design phase?

  • Reduce long, overlapping, noisy prompt lists (a single AIS Topic can easily span 60–200 prompts).
  • Make it easier to organise prompts by grouping them into clear Intent Themes.
  • Ensure we cover the full set of “jobs to be done” within each Topic.
  • Provide a clean way to prioritise which questions enter Phase‑1 vs later phases.

4.5 Branded vs Non‑Branded Prompts (AIS)

Branded prompts

  • Include “AIS”, “AIS Fibre”, “myAIS”, specific plan or package names.
  • Best for:
    • Policy clarity (fees, contract terms, speed, fair use, cancellation conditions).
    • Brand and safety (network quality, security, spam, fraud).
    • Product‑specific comparisons (e.g. between AIS packages or AIS vs competitors).

Non‑branded prompts

  • Generic but relevant, e.g. “unlimited mobile data plan thailand”, “how to avoid roaming bill shock”.
  • Useful where AI answers often compare multiple operators and we want AIS to be correctly represented.

5. The 4‑Step Prompt Design Framework

We’ll use a simple, repeatable loop tailored to AIS:

  1. Start from Topics
    Use Categories to define clear, customer‑facing Topics as hypotheses.
  2. Harvest real questions
    For each Topic, pull questions from GSC, on‑site search, myAIS search, support/chat logs, FAQ & Help.
  3. Normalise into Prompts
    Clean, tag, cluster and score questions (user demand and business impact, 1–3). Promote the best into a curated prompt set per Topic.
  4. Iterate using LLMO insights
    Watch how prompts perform; add, adjust, retire and refine prompts over time.

6. Step 1 – Start from Topics

6.1 Sample Topics for AIS

Goal: define Topics that reflect AIS journeys and organisation. We start from Categories and propose Topics as hypotheses.

Illustrative Categories & Topics

  • Category: Mobile plans & usage
    • Topic: Prepaid top‑up & data packages.
    • Topic: Postpaid bills, data usage & limits.
    • Topic: SIM activation, number transfer (MNP) & new sign‑ups.
  • Category: AIS Fibre broadband
    • Topic: Availability, speed options & eligibility.
    • Topic: Installation, appointment booking & rescheduling.
    • Topic: Connection issues, speed problems & troubleshooting.
  • Category: Roaming & international
    • Topic: Roaming package selection & activation.
    • Topic: Roaming rates, bill shock & usage monitoring.
  • Category: Billing & payments
    • Topic: Understanding bills, charges & due dates.
    • Topic: Payment methods, auto‑pay & failed payments.
    • Topic: Refunds, adjustments & disputes.
  • Category: Rewards & loyalty
    • Topic: AIS Points earning & redemption.
    • Topic: Promotions, bundles & special offers.

6.2 Why Topics Come Before Prompts

If we jump straight to prompts:

  • Lists get long, overlapping and noisy.
  • It’s hard to know if we’ve covered the full journey (e.g. only focusing on billing but not on upgrades or cancellation).
  • Ownership and KPIs per area become unclear (who owns which prompts?).

Starting from Topics ensures:

  • Prompts are anchored in AIS customer journeys (e.g. sign up, use, pay, troubleshoot, renew).
  • We can assign owners, attach KPIs and plan depth per Topic.
  • Later prompt lists are manageable, explainable and easier to govern.

6.3 Designing “Good” Topics

Good Topics are:

  • Aligned to customer journeys – they reflect what customers are trying to do:
    • “Fix my slow home internet.”
    • “Understand and pay my mobile bill.”
    • “Use roaming without risk of bill shock.”
  • Ownable by a team – one area (e.g. Mobile Postpaid, AIS Fibre, Roaming, Digital, Care) can own content, policy and KPIs for the Topic.
  • Mid‑level in granularity – big enough for multiple related intents, but small enough to be meaningful in dashboards and reviews.

7. Step 2 – Harvest Real Questions

7.1 Harvest Real Questions per Topic

For each Topic, we collect question patterns from:

  • GSC / external search – queries for that Topic where AIS or ais.th appears.
  • On‑site & in‑app search – internal search terms in ais.th and myAIS.
  • Support & chatbot logs – repeated phrases and complaint patterns.
  • FAQ & Help – Q&A pages mapped to the Topic, used to validate wording and coverage.

7.2 Example Harvest – “Roaming rates & activation”

Category: Roaming & international
Topic: Roaming rates & activation

From data (and optionally PAA/LLM) we might see questions like:

  • “How do I activate AIS roaming before travelling abroad?”
  • “What are AIS roaming charges for Japan?”
  • “Can I limit my roaming data to avoid bill shock?”
  • “Can I buy AIS roaming packages from the myAIS app?”
  • “Why is my roaming not working when I arrive?”

7.3 Validate and Refine Topics Using Real Questions

Once questions are harvested, we validate and refine our Topics:

  • Review Topics from Step 1 – compare against key journeys and owner teams.
  • Test with real questions – do GSC / onsite / support / FAQ questions fit naturally under this Topic?
  • Split or merge as needed:
    • If a Topic feels crowded or messy → split into 2–3 clearer Topics.
    • If two Topics always share questions → merge them.
  • Use customer‑friendly names – name Topics using phrases AIS customers would recognise, not internal jargon.

8. Step 3 – Normalise into Prompts

8.1 Clean, Tag and Cluster

1. Clean the pool

Remove:

  • Obvious noise (test strings, junk queries).
  • Out‑of‑scope questions (other brands, other countries, non‑telecom topics).
  • Pure navigation (e.g. “ais login”, “myais app download”).

2. Tag and cluster

  • Tag each question with:
    • Source (GSC, onsite search, app search, support, chatbot).
    • Frequency (High / Medium / Low).
    • Journey stage (Pre‑purchase, Onboarding, Usage, Support, Renewal/Churn).
  • Cluster similar questions into single intents.

8.2 Canonical Prompts with Evidence

For each cluster of questions with the same intent, define one canonical prompt in AIS customer language, plus evidence behind it.

Example

  • Category: AIS Fibre broadband
  • Topic: Installation, appointment booking & rescheduling
  • Canonical prompt: “How long does AIS Fibre installation take and can I reschedule my appointment?”
  • Evidence tags (internal):
    • Sources: GSC + Contact centre + myAIS search.
    • Frequency: High.
    • Journey: Onboarding · Installation.

8.3 Scoring User Demand & Business Impact

Goal: decide how frequently each intent comes up, and how important it is for AIS.

Demand score (1–3)

  • 3 – High demand
    • High frequency in GSC and/or on‑site/app search, and/or
    • Recurs clearly in support/chat logs.
  • 2 – Medium demand
    • Shows up repeatedly, but not at the very top.
  • 1 – Low demand
    • Rare / niche, only occasional mentions.

Impact score (1–3)

  • Based on:
    • Risk (bill shock, churn, regulatory exposure).
    • Value (ARPU, upsell/cross‑sell, AIS Points engagement).
    • Experience (NPS, complaints, repeat contacts).

8.4 Selecting Curated Prompt Sets per Topic

Use Demand × Impact to prioritise:

  • Focus Phase‑1 on prompts with:
    • Demand ≥ 2 and Impact ≥ 2, especially the 3×3 cluster (high demand, high impact).
  • Defer or exclude low‑demand, low‑impact prompts to keep the Phase‑1 footprint focused and manageable.

8.5 Internal Design Sheet vs LLMO Sheet

Internal design sheet (for AIS teams)

  • Fields: Category · Topic · Intent Theme (optional) · Prompt · Demand · Impact · Sources · Notes.

LLMO CSV (upload‑ready)

  • Fields: Category · Topic · Prompt · Market (e.g. TH).
  • Demand/impact scores and Intent Themes stay in internal design sheets, not as LLMO fields.

9. Step 4 – Iterate using LLMO Insights

9.1 Using LLMO to Refine Topics & Prompts

Once prompts are live, LLMO shows for each prompt:

  • Visibility – how often AIS appears in AI answers across models.
  • Citations – which URLs are referenced (ais.th vs competitors vs third‑party sites).
  • Trends – whether AIS’ presence is improving, flat or declining.

We use these signals to:

  • Confirm which prompts are working well.
  • Identify high‑impact prompts with weak visibility or weak citations.
  • Spot new question patterns that may deserve new prompts or Topic refinements.

9.2 Prompt Lifecycle: Protect, Optimise, Retire

  • Protect
    • High demand, high impact prompts with strong visibility and accurate AIS citations.
  • Optimise
    • High demand/high impact prompts where:
      • AIS is not cited as often as competitors or third‑party sites.
      • Answers are incomplete, outdated or unclear.
  • Retire
    • Low demand/low impact prompts, or prompts where:
      • LLMO repeatedly shows stable, strong coverage and we want to reduce noise.

9.3 When to Add New Prompts (Inside Existing Topics)

  • Uncaptured high‑demand queries – strong volume in search/support logs not yet added as prompts.
  • Distinct intents hidden in one prompt – LLM answers look confused or mix journeys (e.g. roaming activation + billing) → split into separate prompts.
  • New ways to ask about existing journeys – new channels, devices, payment methods or propositions that appear often enough to matter (e.g. e‑wallet payments, 5G‑specific questions, new AIS bundles).

9.4 When to Add or Split Topics

  • Overloaded Topics with mixed behaviour
    • A Topic carries 60–200 prompts with very different answer patterns.
    • Clear sub‑clusters by intent, owner or journey (e.g. delivery vs refunds vs billing in one Topic).
  • High‑impact clusters buried inside a broad Topic
    • A group of prompts shares language, journeys and KPIs, and shows high demand/impact.
    • That cluster has different risk or ownership (e.g. financial hardship, fraud/scams, network outages).
  • New strategic or product areas
    • LLMO shows rising demand around new AIS propositions (e.g. new 5G packages, converged offers, new loyalty benefits).
    • These questions need their own Topic for tracking and clear ownership.

10. From Framework to Execution

10.1 Phase‑1 Prompt Footprint

Phase‑1 prompt footprint (proposal, not a standard):

  • Target ~1,500 prompts across all Topics.
  • For a large telco like AIS, ~1,500 prompts is a sensible starting point; range ~1,200–2,000 depending on capacity and ambition.

Illustrative Phase‑1 allocation (to be adapted with AIS)

  • Mobile plans & usage: ~500–700 prompts.
  • AIS Fibre broadband: ~300–450 prompts.
  • Roaming & international: ~200–300 prompts.
  • Billing & payments: ~150–250 prompts.
  • Rewards & loyalty + other journeys: remaining prompts.

10.2 Applying the 4‑Step Framework in Phase‑1

To deliver Phase‑1:

  • Confirm scope and ownership
    • Agree Categories, Topics and optional Intent Themes.
    • Assign owners per Topic (Mobile, AIS Fibre, Roaming, Billing, Digital, Care, etc.).
  • Build and validate the design sheet
    • Harvest questions, cluster into prompts, score Demand & Impact.
    • Review with Topic owners and refine.
  • Prepare the LLMO CSV
    • Keep Category/Topic names consistent.
    • One clear question per row; remove duplicates.

10.3 Practical Configuration & Governance

Configuration (what goes into LLMO)

  • Each prompt in LLMO has: Category · Topic · Prompt · Market (e.g. TH).
  • Demand/impact scores and Intent Themes stay in internal design sheets, not as LLMO fields.

CSV / sheet hygiene

  • Keep Category/Topic names consistent across design and upload sheets.
  • If using Intent Themes, keep them in a separate column (design sheet only).
  • Remove duplicates and ensure one clear question per row.

10.4 Scaling Beyond Phase‑1

  • Depth before breadth
    • First, add prompts to high‑value Topics (e.g. bills, roaming, AIS Fibre issues) where LLMO shows gaps.
  • Data‑led expansion
    • Use LLMO visibility, citations and trends, plus demand/impact scores, to decide:
      • Which intents need more coverage.
      • Which new Topics deserve prompts.

11. Recap & Decisions

11.1 Recap

  • Shared vocabulary: Category → Topic → (optional Intent Theme) → Prompt.
  • 4‑step Topic & Prompt framework:
    1. Start from Topics.
    2. Harvest real questions (GSC, onsite/app search, support, chatbot; FAQ/Help as curated inputs; PAA/LLM as optional enrichers).
    3. Normalise into prompts and score user demand and business impact (1–3).
    4. Iterate using LLMO insights and scale towards ~1,500 prompts for AIS.

11.2 Suggested Decisions for AIS

  • Agree initial Categories & Topics to include in Phase‑1.
  • Nominate Topic owners across AIS teams.
  • Align on a Phase‑1 target footprint (e.g. ~1,500 prompts) and a timeline.
  • Define a governance and review cadence (e.g. quarterly prompt review using LLMO insights).

Next steps can include: building the first AIS design sheet, defining owners per Category/Topic, and running a pilot upload into LLMO for 1–2 journeys.
