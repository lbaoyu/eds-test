Adobe LLM Optimizer – 指標解讀與操作建議

回覆說明

本文件整理了針對 Adobe LLM Optimizer 的常見問題與建議解讀，重點放在實務上的解釋與操作建議，協助團隊以這些數據做決策與後續優化。

一、Visibility Score 數據解讀

1-1. 為什麼同一期間 AI mode / AIO / Gemini 差異這麼大？

LLM Optimizer 會針對每一個「模型／介面」分別量測，例如：

  • Google Search AI mode（搜尋整合的 AI 結果）
  • Google AIO / Search Generative Experience
  • Gemini App / 網頁版 Gemini

這些雖然都來自 Google 生態系，但實際上：

  • 使用的索引、檢索邏輯、介面呈現並不完全相同
  • 有些更偏「搜尋輔助」，有些偏「對話式助理」

例如 3/23 若出現 AI mode 23% / AIO 19% / Gemini 42%，代表的是：

在同樣的提示語（prompt）下，Gemini 回覆中對 CUB 的提及、引用與排序表現，比 AI mode / AIO 更好， 而不是系統出錯，也不表示 Google 後端使用完全相同的資料與邏輯。

建議解讀方式：

  • 同一模型內看「趨勢」：例如 Gemini 是否持續上升或下降
  • 同一模型內看「與競品比較」：例如 Share of Voice / Market Comparison
  • 模型與模型之間的差異，視為不同「渠道」的表現，而不是期待數值要靠攏

1-2. Visibility Score 算式與 Mentions 權重

Visibility Score 的概念性公式如下：

35% × Mentions + 15% × Citations + 30% × Position + 20% × Sentiment

(i) Mentions 是否是最重要的關鍵？

  • 從權重來看，Mentions（35%）確實是單一權重最高的因子。
  • 但 Position（排序）+ Citations + Sentiment 合計仍佔 65%。
  • 實務上，如果只有「提到品牌」但：
    • 沒有連結到 CUB 網站（Citation 低）
    • 品牌名稱出現在答案很後面（Position 差）
    • 語氣偏中立甚至負向（Sentiment 不好）

那 Visibility Score 仍然不會高。

可以給客戶的總結說法：

Mentions 是 Visibility 的「前提條件」，沒有被提到就談不上可見度，但要拿到高分，仍需同時兼顧「有被引用、有靠前、語氣正向」。

(ii) Sentiment 長條圖怎麼轉成 20% 的分數？只看 Positive 比例嗎？

介面上的 Sentiment distribution（Positive / Neutral / Negative）是用來直觀顯示各情緒比例。 在實際計算 Sentiment 子分數時：

  • 系統會綜合考慮 Positive、Neutral、Negative 三種比例，而不是只看 Positive。
  • 可以這樣直覺理解：
    • Positive 越高、Negative 越低 → Sentiment 子分數越接近滿分
    • 大部分回答都 Neutral → Sentiment 子分數中等
    • 出現明顯 Negative → Sentiment 子分數會被拉低

可對外溝通的說法：

Sentiment Score 並不是單純的「Positive 百分比」，而是依照 Positive / Neutral / Negative 的分布來換算。趨勢上，只要 Positive 比例升高、Negative 降低，就會拉高 Visibility 裡的 Sentiment 那一塊。

(iii) Position 在介面哪裡？是 Market Comparison 嗎？

  • Position 是「在每個 LLM 答案中，CUB 被引用／提及的平均排名」（越前面越好）。
  • 目前 GA 介面中沒有獨立欄位顯示 Position 數字，而是：
    • 被折算進 Visibility Score
    • 影響部分圖表與比較視圖中的排序

Market Comparison / Share of Voice 則是：

  • 在同一組 prompt 下，CUB 與其他銀行品牌（Other Brands）之間「被提及／被引用」的相對占比。
  • 概念是品牌之間的「聲量和市佔」，而不是單一回答中的排名。

可對外溝通的說法：

Position 是 Visibility Score 的一個內部構成因子；畫面上的 Market Comparison 則是不同品牌之間的「市佔對比」，兩者不是同一個指標。

(iv) 四個變數各自怎麼提升？彼此有關聯嗎？

可以用下表跟團隊對齊：

因子 意義 典型提升手段（結合 LLMO 功能） 與其他因子的關係
Mentions (35%) 在回答文字中有沒有提到 CUB ① 在自家站上針對該主題有足夠內容（FAQ、產品比較）
② 在 LLM 常引用的第三方（維基、評測、新聞、論壇）增加品牌露出
③ 在 LLMO 中設定好品牌別名（Brand aliases） 		Citations 增加通常會推升 Mentions，但不是一對一；有時 LLM 只描述品牌、不給連結。
Citations (15%) 回答中實際引用 CUB 網站 URL ① 用 LLM Visibility Checker 檢查關鍵頁面的「可讀性」
② 使用 Opportunities 裡的建議（Recover hidden content、Unblock agent access、Fix error pages），改善 robots.txt、JS 過重、4xx/5xx 等問題
③ 加上結構化資料（FAQ / schema） 		Citations 提高通常有助於 Position（讓內容被放在答案較上方）。
Position (30%) 在回答中的出現順位 讓 LLM 覺得「CUB 是該主題最相關、最權威的來源」，包含：
① 內容完整且容易被摘要
② 技術可讀性佳（LLM 看得到主要文字）
③ 傳統 SEO 排名與權威度 		Mentions + Citations + SEO / 權威度 → 共同決定 Position。
Sentiment (20%) LLM 描述 CUB 的語氣是正面／中立／負面 ① 自家內容對產品風險、費率、優勢說明清楚，減少 LLM 猜測
② 持續關注第三方評價，若有負面事件，及時提供最新、平衡的說明內容給 LLM 參考 		若負面新聞多，就算 Mentions 高也可能拉低 Visibility；正向案例與清楚說明可彌補。

關於「Citations 多，Mentions 就一定多嗎？」

  • 多數案例中，「技術可讀性提升 → Citations 上升 → Mentions 也常跟著增加」。
  • 但 LLM 行為有噪音且常更新，因此兩者並非線性關係。

實務建議是：不要只追某一項，而是先修技術（讓 LLM 看得懂）、再從內容與外部聲量切入。

二、Sentiment distribution：Positive 合理值？Gemini 比較正向、GPT 比較中立怎麼看？

1. 沒有「通用標準值」

  • 各產業、各主題的「合理 Positive 比例」都不同，例如：
    • 信用卡費用說明類問題，LLM 往往採「中立、說明風險為主」。
    • 品牌形象或 CSR 類問題，Positive 可能較高。
  • 真正有意義的是：
    • 跟自己過去比：Positive 是否比上個月高？Negative 是否被壓低？
    • 跟競品比：同一個 prompt、同一個 LLM 下，CUB 的 Sentiment 和其他銀行相比如何？

2. 為什麼 Gemini Positive 普遍較高，而 GPT (paid) 較多 Neutral？

這多半反映不同 LLM 的「回覆風格與安全政策」：

  • 有的模型在金融建議上更保守，偏向中立語氣並附上警語。
  • 有的模型則使用較多正向形容詞（friendly / helpful tone）。

這不代表「某模型比較偏愛 CUB」，比較接近：

在這組問題上，Gemini 的語氣相對樂觀，GPT 的語氣相對中性。

建議操作：

  1. 抽幾個代表性 prompt，實際打開 Gemini / GPT 的答案：
    • 確認內容是否真的「更推薦 CUB」，還是只是字面語氣較熱情。
  2. 無論哪個模型，都希望達到：
    • 對 CUB 產品的描述正確、完整、不誤導。
    • 有清楚說明風險與比較點，避免未來監管或客訴風險。

三、Citation：Gemini 幾乎一直是 0？整體 Citation 偏低如何解讀與改善？

3-1. 為何 Gemini Citation 長期為 0？

可能原因有幾類，並不只是網站架構單一因素：

  1. Gemini 回答的呈現型態
    • 有些 Gemini 介面會給出長段文字說明，但不一定附上可被偵測的「超連結引用」。
    • 這種情況下，即使實際上參考了 CUB 的內容，在 LLM Optimizer 裡仍會看到 Citation = 0。
  2. 引用的是第三方網站，而不是 cub.com.tw 或相關官方網站
    • 例如新聞、比較網站、論壇或政府單位頁面。
    • 這會讓「CUB 有被提到（Mentions）」，但 Citation 落在外部網站上。
  3. 技術可讀性或存取問題
    • 若關鍵內容高度依賴 JavaScript 渲染、內容藏在互動元件後（需點擊才顯示）、或被 robots.txt / CDN 規則誤擋，LLM 可能不容易選到這些頁面作為引用來源。
    • 這種情況可透過 LLM Visibility Checker 外掛與 LLM Optimizer 內的 Opportunities（例如 Recover hidden content、Unblock agent access）來診斷。

整體而言，這通常不是「因為某個特定網站結構一定阻擋 Gemini」，而是需要綜合評估：

  • 關鍵產品頁面對 LLM 的可讀性與技術狀態。
  • 在同一主題下，LLM 目前偏好引用哪些網站來源。

3-2. Citation 普遍不高，如何善用 dashboard 解讀與改善？

建議的實戰流程：

  1. 在 Brand Presence 裡鎖定高價值 Topic
    • 例如信用卡核卡、回饋、跨境刷卡等。
    • 在這些 Topic 下，篩出「Prompts popularity 高、但 Citation 低」的 prompt。
  2. 查看 URL Inspector / Opportunities（若已串 CDN logs）
    • 確認哪些 CUB URL 有被 LLM 爬過，但 Citation 仍然偏低。
    • Opportunities 中會列出：
      • Recover hidden content（JS 過重、內容沒被看到）。
      • Unblock agent access（誤擋 LLM bots）。
      • Fix error pages（4xx / 5xx）。
      • Add FAQs / Enhance summarization（內容有但不易被摘要）。
  3. 分析外部來源（Earned / Social & Community）
    • Dashboard 中會顯示「常被 LLM 引用的第三方網站」。
    • 若長期都引用某個比較網站或媒體而很少引用 CUB：
      • 可評估是否爭取官方合作內容。
      • 由 CUB 提供更完整的產品介紹／FAQ。
      • 或優化 CUB 自己在維基等公開來源上的介紹。

四、Prompts 設計與 popularity

4-1. 類別階層設定：Category / Topic / Prompts 的建議

原本「Category = 個人金融；Topic = 產品；Prompts = 產品下細部問題」的方向是對的，但可以稍微調整，讓後續分析更有用。

實務建議（以銀行為例）：

  • Category（大類）
    建議對應「業務線／大產品族群」，例如：
    • 個人金融、信用卡、房貸、數位銀行、財富管理。
  • Topic（主題 / 情境 / 子旅程）
    每個 Category 下，再依「客戶問題類型」切分，例如：
    • 信用卡：核卡資格、年費與免年費條件、點數與回饋、海外刷卡 / 手續費、權益比較（機場接送、保險等）。
    • 數位帳戶：開戶資格、利率與門檻、轉帳與跨行手續費、綁定行動支付。
  • Prompts（具體問句）
    • 使用「真人會問的完整句子」，同時包含 branded + unbranded 問句，例如：
      • 「國泰世華哪一張信用卡海外刷卡回饋最高？」（branded）。
      • 「出國刷卡怎麼選現金回饋信用卡？」（unbranded）。
    • 每個 Topic 下有約 20–50 個 prompts 會比較理想，兼顧廣度與可管理性。

可以把 prompts 想成「SEO 關鍵字」，但以自然語言問句的方式呈現。

4-2. 用 Excel template 上傳後資料較難閱讀的常見原因

  1. Category / Topic 命名過長、過細
    • 建議使用短而明確的命名，例如「信用卡-核卡資格」「信用卡-海外刷卡」。
    • 避免 Category 是一整句話，或 Topic 同時包含太多子概念。
  2. 單一 Topic 下塞太多 prompts（上百條）
    • 介面上會變成非常長的列表，不利於分析。
    • 建議拆成多個 Topic，例如：
      • 信用卡-核卡資格（一般）。
      • 信用卡-核卡資格（學生族群）。
      • 信用卡-核卡資格（高資產）。

建議未來仍可用 Excel 進行「大量建立」，但要維持Category / Topic 結構清楚，每個 Topic 的 prompt 數量適中，方便後續做 Topic-level 分析。

4-3. Prompts popularity 都不高，要刻意拉到 High 嗎？有沒有功能幫忙選應該追哪個 prompt？

在產品中，「Prompts popularity」的主要用途是顯示每個 prompt 大致上的需求／熱度，其判斷可能綜合系統估算及（若有串接）Search Console 等來源。

幾個關鍵觀念：

  1. Popularity 不是可以直接「優化」的指標
    • 它不是「LLM 被問了多少次」的精確計數。
    • 也不是系統會自動幫你「把 popularity 拉高」。
    • 而是用來判斷：這個問題本身在市場上有沒有量、熱度如何。
  2. 若目前看到多數 prompt 的 popularity 偏低，可能代表：
    • 上傳的 prompts 偏細、偏內部用語。
    • 或是多數屬於「低搜尋量」題目。
  3. 是否需要在意 popularity？
    • 高 popularity + 高商業價值 → 一定要追、要優化 Visibility。
    • 低 popularity 的 prompt：
      • 若是內部非常重視的關鍵議題，仍可保留觀察。
      • 若重要性較低，可以當作補充，不必投入太多優化資源。
  4. 協助找出高價值 prompts 的方式
    • LLM Optimizer 可協助產生建議 prompts（AI 生成）。
    • 與 Search Console 整合後，可根據真實搜尋需求推薦高價值 prompts。
    • 同時也建議參考：
      • Google Search Console。
      • 搜尋結果中的「人們也會問」(People also ask)。
      • 現有官網 FAQ / Call Center FAQ。
      來反推哪些問題應該變成 prompts。

五、如何判斷要改內容還是改 prompt？及各種組合怎麼解讀？

5-1. Prompts popularity × Visibility score 的決策表

可以用下表跟團隊對齊策略：

Popularity Visibility 解讀與建議
低 低 多半代表「市場詢問量小 + CUB 也沒被看到」。若主題本身商業價值不高，可以暫時不投資； 若主題很關鍵，可能是 prompt 寫法太「內部話」，建議： ① 改寫成更貼近一般人會問的方式； ② 或參考 Search Console 常見關鍵字重寫。
低 高 代表這個問題雖然市場量不大，但在 LLM 裡 CUB 表現其實不錯。建議：保留即可，不必刻意「把 popularity 拉高」，因為那反映的是市場需求，不是可優化指標。
高 低 這是最優先處理的一群：市場很常問，但 CUB 可見度低。建議： ① 先檢查是否有對應的產品／FAQ 內容； ② 用 Visibility Checker 檢查關鍵頁面； ③ 查看該 prompt 下 LLM 主要引用哪些競品或第三方，作為 GEO 優化重點。
高 高 目前的 sweet spot：市場常問且 CUB 表現好。建議： 持續監控 Sentiment 與競品表現，並把這些成功案例當作「模板」，延伸出類似的 prompt 與內容策略。

什麼時候要「改內容」 vs. 「改 prompt」？

  • 建議原則：
    1. 先觀察至少 2–4 週數據（LLM 本身波動很大，一週的變化可能只是雜訊）。
    2. 若某 prompt popularity 高，且對應的業務場景重要：
      • 優先檢查內容與技術面（是否有對應落地頁、FAQ、技術可讀性是否良好）。
    3. 若 prompt 寫法看起來不像「一般人會問的語句」：
      • 先在 Google / ChatGPT 等實際輸入類似問題，觀察常見問法。
      • 再依實際搜尋建議與「人們也會問」重寫 prompt 內容。

5-2. 為什麼不同工具在同期間變化很懸殊？未來會有跨模型綜合數據嗎？

例如 12 月以來不同模型數字落差（ChatGPT Free 從 41% → 17%；Gemini 從 41% → 48%）是目前多數品牌都在面臨的情況。

主要原因：

  • 各家 LLM 持續：
    • 更新模型版本。
    • 調整搜尋／引用策略。
    • 改變介面中的引用呈現方式。
  • 這些變化通常不會同步發生在每個模型上，因此：
    • ChatGPT 可能在某一週大量改版，導致 Visibility 下降。
    • 同時 Gemini 針對某類金融內容加強引用某些資訊源，反而讓 CUB 在 Gemini 上提升。

建議解讀原則：

  1. 不要只看單一週的變化
    建議至少看 4–8 週趨勢，再判斷是「趨勢」還是「雜訊」。
  2. 先分模型看，再看「跨模型共識」
    • 若 ChatGPT / Gemini / Copilot 都同時往下掉 → 較可能是 CUB 自身內容或外部聲量出現變化。
    • 若只有某一個模型掉、其他持平甚至上升 → 多半是該模型本身調整或偏好來源改變。
  3. 搭配 URL Inspector / Opportunities 看「掉到哪裡去了」
    • 例如檢查 ChatGPT 的 Citation 是否從 CUB 網站轉向某個比較網站或新聞。
    • 這些資訊可直接轉化為外部管道與內容策略的優化線索。

未來是否能有「跨模型綜合數據」？

  • 目前產品已提供：
    • Brand Presence 頁面頂端的 All engines 聚合視圖（全部 LLM 的 Mentions / Citations 總和趨勢）。
  • 產品路線圖中也規劃：
    • 更完整的「Executive summary / 跨模型匯總」視圖，協助整合不同 LLM 的 Visibility、Sentiment、Share of Voice。
    • 以一頁報表方式，讓管理層快速掌握整體 AI 可見度狀況。

對外可說明為：

目前已可在介面中看到「所有模型加總」的指標；更完整的跨模型匯總與高階報表也在產品路線圖中，將會逐步釋出。

六、系統操作／介面問題

6-1. Other Brands 的地區要更新，只能刪掉重建嗎？

目前版本的操作建議：

  • 若要調整競品品牌的「地區 / Region」設定，最穩定的作法是：
    1. 在「Configuration」裡新增一個新的 Other Brand，選擇正確 Region。
    2. 確認數據開始累積後，將舊的設定刪除或停用。

目前尚未提供直接在原有設定上 inline 編輯地區的功能，因此建議以「新增正確版本，再移除舊版本」的方式操作。這類操作需求會持續回饋給產品團隊作為未來 UX 優化參考。

6-2. 試用版（Try Before Buy）是否建議連動 Google Search Console？可以整合什麼數據？

非常建議即使在試用期間也先串接 Search Console，主要有三個原因：

  1. 更聰明地產生與優先排序 prompts
    • LLM Optimizer 可利用 Search Console 提供的查詢字詞（queries）、登入頁面（landing pages）、曝光與點擊數等資訊。
    • 據此推薦：
      • 哪些搜尋詞應該變成 prompts。
      • 哪些 Topic / Category 的需求量最大。
  2. 讓 GEO（AI 搜尋優化）與 SEO 對齊
    • 可比對：
      • 某 search query 在 SEO 上的表現（曝光 / 點擊）。
      • 對應 prompt 在 LLM 裡的 Visibility / Citations 表現。
    • 幫助找出：
      • 「SEO 很強但 LLM 表現弱」→ 是 GEO 優化機會。
      • 「LLM 表現好但 SEO 弱」→ 可以反向強化傳統搜尋與網站內容。
  3. Search Console 提供的是彙總搜尋數據，不涉及個資
    • 串接在隱私上的風險相對較低。
    • 但對於選題與 prompt 策略優先順序非常有幫助。

可對外建議的說法：

即使在 200 prompts 的試用階段，也建議盡快把 Search Console 串接起來（目前尚未啟用的情況下），這樣在選題與調整 Category / Topic / prompt 時，就能根據真實搜尋需求做決策，而不是只靠體感。

最後小結：建議的「下一步行動」

可將近期重點聚焦在以下幾件事：

  1. 重新整理 prompts 結構
    • 以「信用卡／數位帳戶等 Category × 旅程 / 情境 Topic」為主幹。
    • 對高商業價值主題補足 20–50 個「真人會問」的問句（含 branded + unbranded）。
  2. 鎖定「Popularity 高 × Visibility 低」的組合
    • 利用 LLM Visibility Checker 與 Opportunities 找出技術與內容缺口。
    • 優先從 2–3 個最重要的產品場景著手（例如主力信用卡）。
  3. 儘速完成 Google Search Console 串接
    • 以真實搜尋需求來微調 prompts 與 Topic 優先序。
    • 同步觀察 SEO × LLM Visibility 的差異，找出 GEO 的最快勝點。
  4. 持續進行「Optimize at Edge」試用
    • 維持現行的 Optimize at Edge 試用與實驗設定，在流量與風險可控的前提下，逐步擴大涵蓋的頁面與體驗。
    • 觀察 Edge 端優化對 LLM Visibility、SEO 與轉換率的實際影響，優先保留表現佳的版本作為長期方案。
